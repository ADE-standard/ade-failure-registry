# Agent Delivery Engineering (ADE) Failure Taxonomy — 五层失效分类学

> 从三篇 ADE 论文提炼的统一分类框架，适用于所有 AI Agent 架构。

## 设计原则

1. **跨架构**：分类基于失效的本质机制，不绑定任何特定 Agent 框架
2. **多层递进**：从物理通道到认知行为，逐层抽象
3. **MECE**：互斥且穷尽，每个案例只归一个位置
4. **可验证**：每个分类有明确的判定标准和检测方法

---

## L1 — 通道层 (Channel Layer)

> 论文来源：[Channel Fracture](https://arxiv.org/abs/2606.04896)

**定义**：系统间通信通道断裂，但系统本身仍在运行。

### L1-A: 发送端断裂 (Sender-side Fracture)

通道从发送端断开。发送方认为已发送，但消息从未到达。

| 判定标准 | 检测方法 |
|----------|----------|
| 发送方报告成功 | 检查发送方日志 |
| 接收方从未收到 | 检查接收方日志 |
| 无错误传播 | 发送方无 error 日志 |

**典型案例**：
- Email Gateway 自动发送（发送成功但违反策略）
- cron agent memory 写入（skip_memory=True 静默跳过）

### L1-B: 接收端断裂 (Receiver-side Fracture)

通道从接收端断开。发送方发出的消息无法被接收方处理。

| 判定标准 | 检测方法 |
|----------|----------|
| 发送方报告成功 | 检查发送方日志 |
| 接收方连接已断 | 检查接收方连接状态 |
| 无重连触发 | 接收方无 reconnection 日志 |

**典型案例**：
- WeCom 846609：send 路径收到断连信号但不触发重连
- QQ Bot 僵尸连接：服务端已断开，客户端不知情

### L1-C: 状态同步断裂 (State Sync Fracture)

双端各自运行正常，但状态不一致。

| 判定标准 | 检测方法 |
|----------|----------|
| 两端各自报告正常 | 分别检查两端日志 |
| 状态值不匹配 | 对比两端状态记录 |
| 无一致性校验 | 无 cross-check 机制 |

**典型案例**：
- WeCom ACK 丢失导致重复消息
- Desktop UI 与磁盘状态不一致

## L2 — 可观测层 (Observability Layer)

> 论文来源：[Silent Failure](https://arxiv.org/abs/2606.08162)

**定义**：系统发生故障但不产生可观测的告警。

### L2-A: 错误静默 (Silent Error)

故障发生但不报错，或错误信息不描述根因。

| 判定标准 | 检测方法 |
|----------|----------|
| 故障确实发生 | 事后验证 |
| 无 error/warning 日志 | 检查日志 |
| 或日志信息误导 | 日志内容 vs 实际原因 |

**典型案例**：
- WeCom WebSocket 重连失败但无日志
- cron 审批死锁：报 "timeout" 而非 "approval deadlock"

### L2-B: 成功幻觉 (Success Illusion)

操作未实际完成但报告成功。

| 判定标准 | 检测方法 |
|----------|----------|
| 系统报告 success | 检查返回值 |
| 实际效果未达成 | 验证最终状态 |
| 无验证步骤 | 缺少 post-check |

**典型案例**：
- Desktop 删除 profile 报成功但文件仍在
- V4A patch 删除操作对不存在的文件报成功

### L2-C: 恢复机制自身静默失效 (Recovery Mechanism Silent Failure)

为故障恢复设计的机制本身也静默失效——二阶静默失效。

| 判定标准 | 检测方法 |
|----------|----------|
| 存在恢复机制 | 检查代码/配置 |
| 恢复机制未执行或执行失败 | 检查恢复日志 |
| 无告警 | 无 error/warning |

**典型案例**：
- WeCom WebSocket 重连逻辑可能无限挂起
- cron 审批死锁：安全机制在自治上下文中产生死锁

## L3 — 数据层 (Data Layer)

> 论文来源：[Intelligence Entropy](https://arxiv.org/abs/2606.18065)

**定义**：数据在存储、传输或处理过程中发生不可逆的质量退化。

### L3-A: 数据丢失 (Data Loss)

数据被永久删除且无法恢复。

| 判定标准 | 检测方法 |
|----------|----------|
| 数据曾经存在 | 检查历史/备份 |
| 现在不存在 | 验证当前状态 |
| 无恢复路径 | 无 backup/rollback |

**典型案例**：
- Context compression 丢失未 flush 的消息
- V4A patch 对 >2000 行文件静默截断

### L3-B: 数据衰变 (Data Decay)

数据仍在但质量退化。

| 判定标准 | 检测方法 |
|----------|----------|
| 数据仍然存在 | 验证存在性 |
| 数据质量下降 | 对比前后质量 |
| 衰变是渐进的 | 非一次性丢失 |

**典型案例**：
- state.db FTS 索引损坏（数据在但搜不到）
- 长对话中记忆衰减（信息还在但不可用）

### L3-C: 数据不一致 (Data Inconsistency)

同一数据在不同存储位置出现矛盾。

| 判定标准 | 检测方法 |
|----------|----------|
| 同一数据有多份 | 检查多存储位置 |
| 值互相矛盾 | 对比各处数据 |
| 无冲突解决机制 | 无 reconciliation |

**典型案例**：
- UI 状态 vs 磁盘状态不一致
- 内存 vs DB 会话数据不同步

## L4 — 认知层 (Cognitive Layer)

> 来源：ADE 框架理论 + 实际运营发现

**定义**：Agent 的知识、记忆或推理能力出现退化。

### L4-A: 知识断裂 (Knowledge Discontinuity)

跨会话的知识丢失。

| 判定标准 | 检测方法 |
|----------|----------|
| 会话 A 中获得知识 | 检查 A 的记录 |
| 会话 B 中知识丢失 | 检查 B 的记录 |
| 无持久化机制 | 检查 memory 配置 |

**典型案例**：
- cron agent 跨会话记忆丢失（skip_memory=True）
- 新会话不继承旧会话上下文

### L4-B: 认知框架滞后 (Cognitive Framework Lag)

Agent 的知识框架跟不上环境变化。

| 判定标准 | 检测方法 |
|----------|----------|
| 环境已变化 | 检查外部状态 |
| Agent 仍用旧框架 | 检查 Agent 行为 |
| 无更新机制 | 无 auto-update |

**典型案例**：
- 配置文件更新但 Agent 行为未变
- 规则记忆与实际操作不一致

### L4-C: 社会常识缺失 (Social Commons Norm Deficiency)

Agent 缺乏人类通过社会化自然习得的常识。

| 判定标准 | 检测方法 |
|----------|----------|
| 行为违反社会/职业规范 | 人工审查 |
| Agent 不知道有此规范 | 检查训练数据 |
| 无法通过简单规则修复 | 需长期习得 |

**典型案例**：
- 引用链不追溯（知道引用但不检查来源）
- 邮件自动回复客户（缺乏商业礼仪常识）

## L5 — 行为层 (Behavior Layer)

> 来源：ADE 框架理论

**定义**：Agent 的行为偏离预期或违反行为契约。

### L5-A: 行为越界 (Behavior Violation)

Agent 做了不该做的事。

| 判定标准 | 检测方法 |
|----------|----------|
| 行为超出授权范围 | 对比 BCP 契约 |
| 造成外部影响 | 检查副作用 |
| 未经人类审批 | 无 approval 记录 |

**典型案例**：
- Email Gateway 自动回复客户（未授权的外部通信）
- Agent 直接修改源码（违反源码禁触铁律）

### L5-B: 行为缺失 (Behavior Omission)

Agent 该做的事没做。

| 判定标准 | 检测方法 |
|----------|----------|
| 存在明确的行为契约 | 检查 BCP |
| Agent 未执行约定行为 | 对比执行记录 |
| 无告警或解释 | 静默跳过 |

**典型案例**：
- cron agent 该写 memory 但跳过了
- 文件归类时未遵守目录结构

### L5-C: 行为不一致 (Behavior Inconsistency)

同一指令在不同上下文产生不同结果。

| 判定标准 | 检测方法 |
|----------|----------|
| 同一指令多次执行 | 对比执行记录 |
| 结果不一致 | 对比输出 |
| 差异无法预测 | 无规律可循寻 |

**典型案例**：
- V4A patch fuzzy match 阈值过低导致不同文件匹配不同策略
- LLM 概率性导致同指令不同结果

## 分类判定流程

```
新案例 → 
  有物理通道断裂？ → L1 (哪一端？ A/B/C)
  有故障但不报错？ → L2 (哪种？ A/B/C)
  有数据质量退化？ → L3 (哪种？ A/B/C)
  有知识/推理退化？ → L4 (哪种？ A/B/C)
  有行为偏离契约？ → L5 (哪种？ A/B/C)
  多层重叠？ → 归最底层（物理优先）
```

## 严重度分级

| 等级 | 定义 | 响应 |
|------|------|------|
| Critical | 数据永久丢失或对外产生不可逆影响 | 立即修复 |
| High | 功能中断但有恢复路径 | 24h 内修复 |
| Medium | 功能降级但可用 | 计划修复 |
| Low | 体验问题不影响功能 | 记录待修 |
